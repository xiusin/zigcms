# 并发与内存模型专题（实战向）

## 基础
- 并发问题本质：共享状态 + 生命周期/所有权不清。
- 优先策略：减少共享，使用消息传递或不可变数据。

## 常见风险
- 数据竞争：多个执行单元同时读写同一内存。
- ABA/可见性问题：读到过期值或顺序不一致。
- 锁泄漏：异常路径未解锁导致死锁。
- 并发下双重释放/悬空：所有权不清导致重复释放或悬空引用。

## 实践策略
- 锁内短逻辑：缩短临界区，避免重锁。
- 单一所有权转移：跨线程传递 owned 数据，原持有者失效。
- 只读共享：`[]const u8` + 明确生命周期；可用写时复制。
- 守卫释放：`defer lock.unlock()`，覆盖异常路径。
- 原子/无锁：计数、标志用原子，避免锁竞争；注意内存序。

## 进阶/易踩坑
- 在迭代时修改共享容器，导致迭代器失效。
- 锁顺序不一致引发死锁。
- 在线程间传递栈上切片/局部对象，离开作用域即悬空。
- 读写混用无同步，读取到半更新状态。

## 场景建议
- 任务队列：生产者分配、消费者释放，或统一线程池释放。
- 配置快照：写时复制，读路径只读无锁。
- 高并发计数：用原子类型；批量合并计数减少争用。
- 缓存更新：双 buffer/版本号切换，读路径只读。

## 上线前检查清单
- [ ] 锁是否都有对称释放（含异常路径）？
- [ ] 是否存在跨线程传递栈对象/短生命周期切片？
- [ ] 共享容器迭代时是否有并发修改？
- [ ] 是否存在锁顺序不一致的路径？
- [ ] 原子操作的内存序是否满足可见性需求？
- [ ] 资源释放责任在线程间是否清晰（避免双重释放/遗漏）？

## 实践图示（共享状态治理）
```
是否必须共享？
  ├─否 → 消息传递/复制 → 无共享
  └─是 → 明确所有权/只读
          ├─只读共享 → 保证生命周期，避免写
          └─可写共享 → 加锁/原子
                  ├─锁：锁内短逻辑 + defer unlock
                  └─原子：明确内存序，避免竞态

跨线程传递
  ├─owned 数据：原 owner 失效
  └─borrowed 视图：禁止跨生命周期/跨作用域
```
