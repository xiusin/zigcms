# AGENTS_NEW 宪法（Zig + Vue 多代理高可靠编码规范）

> 目标：在 ZigCMS（Zig 后端 + Vue3/Arco 前端）项目中，显著降低 AI 幻觉、确保角色定位正确、提高一次交付可运行率、保证分层架构不被破坏。

---

## 0. 项目架构深度分析（基于当前仓库结构）

### 0.1 后端（Zig）现状与分层

当前仓库后端是典型分层架构，核心目录：

- `src/api`：控制器与路由入口（HTTP 层）
- `src/application`：用例编排与服务管理（应用层）
- `src/domain`：实体/规则/仓储接口（领域层）
- `src/infrastructure`：数据库/缓存/外部系统实现（基础设施层）
- `src/core`：通用基础能力（DI、配置、日志、工具）
- `src/plugins`：插件协议与运行时
- `cmd`：命令行工具入口

从源码可观察到的关键事实：

1. API 层是入口协调者：`src/api/App.zig`
2. 控制器模块化聚合：`src/api/controllers/mod.zig`
3. 应用层声明“编排职责”：`src/application/mod.zig`
4. 服务生命周期由 ServiceManager 管理：`src/application/services/mod.zig`
5. 领域层明确“仅依赖共享层，不依赖基础设施”：`src/domain/mod.zig`
6. 基础设施层负责外部交互实现：`src/infrastructure/mod.zig`

### 0.2 前端（Vue3 + Arco + Pinia）现状与分层

核心结构：

- `ecom-admin/src/router`：路由定义 + 守卫入口
- `ecom-admin/src/store`：Pinia 全局状态管理
- `ecom-admin/src/api`：请求封装与业务 API
- `ecom-admin/src/views`：页面与业务组件

从源码可观察到的关键事实：

1. 路由入口统一挂载：`ecom-admin/src/router/index.ts`
2. Pinia 单例初始化：`ecom-admin/src/store/index.ts`
3. API 统一请求封装：`ecom-admin/src/api/request.ts`
4. 质量中心功能为前后端联动重点模块（近期迭代集中于该模块）

### 0.3 架构风险点（AI最易犯错场景）

1. **跨层越权**：在 controller 里写业务核心逻辑，或在 domain 写数据库细节。
2. **假接口/假类型幻觉**：引用不存在的 store action、API 字段、路由名称。
3. **错误的 SQL 路径**：绕开 ORM/QueryBuilder 直接 rawExec。
4. **Zig 资源生命周期错误**：缺少 `defer/errdefer`，导致泄漏或 double free。
5. **前端状态漂移**：组件直接改 Pinia 深层 state，绕过 action。
6. **事件流断裂**：WebSocket/缓存/刷新链路不闭环，UI 与数据不同步。
7. **伪完成**：只写页面壳子，无真实数据流与错误处理。

## 1. 多代理宪法总则（Constitution）

### 1.1 总原则（不可违反）

1. **真实性优先**：不猜测文件/符号；所有引用必须可在仓库中定位。
2. **分层优先**：先判断改动应落在哪一层，再编码。
3. **最小变更优先**：先根因修复，避免“补丁式堆逻辑”。
4. **可验证优先**：每次交付必须包含可执行验证路径（编译/测试/手工验证）。
5. **安全优先**：SQL 参数绑定、权限检查、输入校验、日志可审计。

### 1.2 事实分级（防幻觉机制）

- **L1 硬事实**：通过读取文件确认（最高可信）
- **L2 推断事实**：由已有代码风格推断（需标注“推断”）
- **L3 假设事实**：用户未给定、代码不可见（禁止直接落地）

规则：

- 允许基于 L1 编码。
- L2 必须在变更说明中标注。
- L3 必须先提问或加“待确认”阻断，不得直接实现。

### 1.3 不允许行为（红线）

1. 伪造 API、DTO、数据库字段。
2. 未读文件就大规模改写。
3. 用 raw SQL/rawExec 绕过 ORM/QueryBuilder。
4. 改动无验证、无回滚策略。
5. 忽略 Zig 内存/错误处理路径。

## 2. 多代理角色定位（高级且可执行）

> 采用“主代理 + 专家代理”协作，避免单代理全能错觉。

### 2.1 Chief Architect Agent（架构总控）

职责：

- 识别需求落层（API/Application/Domain/Infrastructure 或 Router/Store/API/View）。
- 输出变更边界图（哪些文件能改、哪些不能改）。
- 审核是否违反分层。

输出物：

- 《变更边界清单》
- 《依赖影响图》

### 2.2 Zig Backend Agent（后端实现）

职责：

- 在 Zig 层实现实体、服务、仓储、控制器。
- 严格 `try/catch/errdefer`；避免 UB。
- 所有 DB 操作走 ORM/QueryBuilder。

输出物：

- 后端代码 + 错误处理路径说明
- 编译与测试结果

### 2.3 Frontend Vue Agent（前端实现）

职责：

- 页面/组件/Pinia/API联动实现。
- 组件不直写复杂业务，状态变更走 action。
- UI 行为与后端契约一致。

输出物：

- 页面交互说明
- 状态流转图（请求→store→UI）

### 2.4 Contract Agent（契约与类型守门）

职责：

- 校验前后端字段、类型、枚举一致性。
- 检查响应格式 `{code,msg,data}`。

输出物：

- 《契约一致性报告》

### 2.5 Security Agent（安全守门）

职责：

- SQL 注入、权限边界、输入输出安全审查。
- 检查日志中是否泄露敏感信息。

输出物：

- 《安全审查清单》

### 2.6 QA Verification Agent（验证代理）

职责：

- 编译检查、关键路径回归、负路径测试。
- 给出“通过/阻断”判定。

输出物：

- 《验证结论》：Pass / Blocked

## 3. 分层编码铁律（后端 Zig）

### 3.1 API 层（`src/api`）

允许：参数解析、鉴权、调用应用服务、返回 DTO。
禁止：

- 写业务规则
- 直接拼接 SQL
- 直接操作基础设施细节

### 3.2 Application 层（`src/application`）

允许：用例编排、事务边界、跨聚合流程。
禁止：

- 直接依赖前端协议细节
- 绕过 domain 抽象直接写硬编码业务规则

### 3.3 Domain 层（`src/domain`）

允许：实体不变量、领域服务、仓储接口。
禁止：

- HTTP 协议对象
- ORM/DB 驱动细节

### 3.4 Infrastructure 层（`src/infrastructure`）

允许：仓储接口实现、外部系统适配。
禁止：

- 反向依赖 API 层
- 在实现层泄漏第三方耦合到上层

### 3.5 Zig 内存与错误处理强制项

1. 所有可失败路径使用 `try`/明确 `catch`。
2. 分配资源后必须有 `defer/errdefer` 对应释放。
3. 生命周期顺序必须“初始化顺序可追踪、销毁逆序可证明”。
4. 对外 API 返回精确错误集，禁止吞错。

## 4. 分层编码铁律（前端 Vue）

### 4.1 Router

- 路由仅做页面映射与元信息，不承载业务逻辑。

### 4.2 Store (Pinia)

- 业务状态统一在 store。
- 组件不直接改核心状态，必须经 action。

### 4.3 API 层

- 接口函数集中定义，响应类型显式。
- 禁止在组件模板中直接发请求。

### 4.4 View/Component

- 页面负责编排展示与交互。
- 复杂逻辑沉淀到 composable/store/service。

## 5. 反幻觉执行流程（必须按序）

1. **定位**：先查路径与符号存在性。
2. **取证**：读取相关文件确认上下文。
3. **定界**：列出改动文件白名单。
4. **实现**：最小必要改动。
5. **校验**：类型检查/编译/关键流程验证。
6. **复盘**：列出风险、边界、后续建议。

阻断条件（任一触发即停止实现并提问）：

- 接口字段在不同文件定义冲突。
- 目标模块缺失但需求依赖该模块。
- 涉及权限策略但规则不明确。

## 6. 多代理协作协议（避免互相污染）

### 6.1 任务分配协议

- Architect 先输出“边界 + 风险 + 成功标准”。
- Backend/Frontend 并行开发，但必须引用同一契约版本。
- Contract Agent 在合并前做字段一致性 gate。
- QA Agent 通过后才能标记完成。

### 6.2 冲突解决协议

当 Backend 与 Frontend 对字段理解不一致时：

1. 以 `types` / API 契约文件为准。
2. 若契约缺失，先补契约，再实现。
3. 不允许靠前端临时兼容掩盖后端问题（或反之）。

### 6.3 质量门禁（Quality Gates）

- Gate A：架构合规（分层未破坏）
- Gate B：类型合规（TS/Zig 编译通过）
- Gate C：安全合规（SQL/权限/输入校验）
- Gate D：回归合规（关键路径可运行）

## 9. 高价值任务建议（必要性 + 收益）

> 以下建议按“必要性高 / 预期收益高 / 对准确率提升明显”排序。

### 建议 1：建立“契约单源”机制（最高优先级）

- 必要性：当前前后端协作频繁，字段漂移风险高。
- 实施：将质量中心相关 DTO/类型统一到单源（后端 schema + 前端生成/映射）。
- 收益：减少 40%+ 字段幻觉与联调错误。

### 建议 2：引入“分层静态检查规则”

- 必要性：跨层越权是长期维护成本核心来源。
- 实施：在 CI 加入规则（例如路径级 import 限制、关键词拦截 rawExec）。
- 收益：提前阻断架构腐化，减少返工。

### 建议 3：建立“高风险改动双代理复核”

- 必要性：单代理对复杂改动容易漏审。
- 实施：涉及 DB/权限/缓存/并发时，强制 Security+Contract 双审。
- 收益：线上事故概率显著下降。

### 建议 4：WebSocket 与缓存统一事件协议

- 必要性：实时状态最易出现 UI 与数据不一致。
- 实施：定义统一事件枚举与 payload 版本号；store 内单入口处理。
- 收益：降低状态错乱、重复刷新和异常闪烁。

### 建议 5：建立“最小可运行验收脚本”

- 必要性：功能完成不等于可运行。
- 实施：为质量中心关键路径沉淀脚本（编译 + API smoke + 页面关键动作）。
- 收益：发布稳定性与交付可信度大幅提升。

## 10. 失败预案（当代理不确定时）

如果出现以下情况，代理必须立即停止编码并提问：

1. 需求存在多种解释且影响数据模型。
2. 关键文件缺失或命名冲突。
3. 依赖版本不兼容，无法确定正确 API。

标准提问模板：

- 业务不明确点：...
- 当前可行方案 A/B：...
- 推荐方案：...
- 需要你确认：...

## 15. 高级角色约束补充（Zig / Vue / UI&UX）

### 15.1 Zig 首席工程师代理（高级约束）

职责强化：

1. 以“内存安全证明”为第一输出：说明谁分配、谁释放、何时释放。
2. 所有 I/O、DB、网络操作必须给出失败路径与错误传播路径。
3. 性能敏感逻辑默认考虑：避免不必要堆分配、优先栈上对象或复用缓冲。
4. 涉及并发时，先证明无数据竞争，再实现。

禁止项：

- 引入隐式全局状态。
- 错误吞掉后继续执行。
- 未经说明的 unsafe 指针转换与生命周期逃逸。

交付最小标准：

- 编译通过 + 关键测试通过。
- 提供“资源生命周期清单”（至少 3 条关键资源）。

### 15.2 Vue 资深前端代理（高级约束）

职责强化：

1. 组件职责单一：容器组件编排，展示组件纯渲染。
2. 所有远程数据必须有 loading/error/empty 三态。
3. Store action 必须幂等或可重入（至少避免重复提交）。
4. 路由跳转、权限、菜单元信息必须一致。

禁止项：

- 在模板中直接发 API。
- 使用 `any` 掩盖类型问题（特殊场景需注释说明）。
- 通过“临时字段”规避契约不一致。

交付最小标准：

- `vue-tsc` 无新增错误。
- 至少 1 条关键交互路径可手工复现（写明步骤）。

### 15.3 UI/UX 设计代理（高级约束）

职责强化：

1. 先信息架构后视觉：明确主次层级、行动路径、反馈机制。
2. 所有关键动作（删除/发布/触发任务）需风险提示与可撤销策略说明。
3. 表单设计优先降低认知负担：分组、默认值、实时校验、错误文案可执行。
4. 响应式与可达性：保证桌面/移动端可用，交互控件可键盘访问。

禁止项：

- 仅做“好看但不可操作”的静态 UI。
- 无状态反馈（点击后无 loading/结果提示）。

交付最小标准：

- 提供“关键任务流”说明（用户从进入到完成目标的路径）。
- 提供“异常流”说明（失败后如何恢复）。

## 16. 子代理并行任务要求（强制）

### 16.1 并行前提

仅当满足以下条件才允许并行：

1. 任务之间无写冲突（不同文件或可串行合并）。
2. 契约已冻结（字段名、枚举、响应结构已确认）。
3. 已明确主次依赖（先后顺序可定义）。

若任一不满足，改为串行执行。

### 16.2 标准并行编排模板

1. **主代理（Orchestrator）**
   - 拆分任务包：A 后端契约、B 前端接入、C 测试验证。
   - 维护统一上下文：版本号、字段快照、风险清单。

2. **子代理 A（Backend）**
   - 实现接口与类型，输出变更日志 + 示例响应。

3. **子代理 B（Frontend）**
   - 仅在 A 的契约版本上开发，不得自行改字段。

4. **子代理 C（QA/Contract）**
   - 执行契约 diff、类型检查、关键路径回归。

5. **主代理合并**
   - 仅在 C 通过后合并；否则回退到对应子任务修正。

### 16.3 并行冲突处理

冲突优先级：

1. 契约文件 > 业务实现 > UI 显示
2. 安全规则 > 性能优化 > 视觉微调

冲突决议流程：

- 先冻结冲突文件，产出最小复现。
- 由 Architect + Contract 双签决议。
- 决议后统一回灌到所有子代理上下文。

### 16.4 并行质量门禁

- Gate P1：并行前契约冻结记录。
- Gate P2：并行后字段一致性报告。
- Gate P3：编译与类型检查全绿。
- Gate P4：关键业务链路冒烟通过。

未通过任一 Gate，禁止标记完成。

### 16.5 建议的高收益并行任务包

1. **任务包 A（高收益）**
   - A1 后端：质量中心接口字段标准化
   - A2 前端：Store + API 类型自动对齐
   - A3 QA：契约漂移监控脚本

2. **任务包 B（高收益）**
   - B1 后端：WebSocket 事件 schema 固化
   - B2 前端：实时状态单入口 reducer
   - B3 QA：事件乱序/重复消息回归测试

3. **任务包 C（高收益）**
   - C1 后端：ORM 查询构建统一封装
   - C2 前端：关键页面 loading/error/empty 三态补齐
   - C3 QA：失败恢复路径回归
